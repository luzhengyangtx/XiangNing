## 1. 产品概述与目标用户

### 1.1 产品定位
一套专为**便利店、礼品店等小型实体零售店**打造的轻量级库存管理SaaS系统。核心差异化在于**原生对接即时零售平台（如美团闪购、饿了么、京东到家、抖音小时达等）API**，实现“平台来单 → 自动减库存 → 多平台库存同步”，彻底告别手动改库存和超卖。

### 1.2 目标用户
- **便利店 / 礼品店主**：日常管理1~5个在线店铺，SKU 在 200~3000 之间，无专职IT人员。
- **小型连锁加盟商**：2~10 家门店，需要中央管控库存，但又希望每家店有一定自主性。

### 1.3 核心价值
- 消除超卖：顾客下单后，**毫秒级**同步扣减所有平台库存。
- 减少人工：商品信息（标题、主图、详情图、价格）一处维护，自动推送到各平台。
- 透明的失败处理：同步失败时，**呈现具体哪条商品、哪个平台、什么原因**，而不是一句“同步失败”。

## 2. 详细功能列表

### 2.1 库存管理核心
| 功能模块 | 功能点 | 描述 |
|---------|--------|------|
| **实时库存看板** | 库存总览 | 表格形式展示所有商品，关键列：商品名称/SKU、当前库存、各平台库存状态（正常/异常/同步中） |
| | 库存搜索/筛选 | 按商品名称、SKU、分类、库存警戒线搜索和筛选 |
| **手动库存操作** | 手动加减库存 | 可对单个或批量商品做加减库存，并强制记录操作日志（操作人、时间、变更量、原因） |
| | 盘点功能 | 创建盘点单，锁定库存，盘点完成后自动修正差异并生成报告 |
| **自动库存同步** | 来单减库存 | 通过 Webhook / 消息队列 接收平台订单消息，自动扣减对应平台+商品库存，并同步扣减其他平台库存 |
| | 进货加库存 | 支持通过扫码枪（或手动输入）入库，增加库存并同步到各平台 |
| | 多平台库存同步 | 当库存发生任何变化（手动/自动），自动向所有已绑定平台推送最新库存值 |
| **库存策略** | 安全库存警戒 | 可设置每个商品的安全库存线，低于警戒线时首页高亮提醒 |
| | 共享库存/独占库存 | 对于接入多个平台的同一商品，支持两种模式：①共享库存（总库存共享）② 平台独占库存 |

### 2.2 商品信息同步
| 功能模块 | 功能点 | 描述 |
|---------|--------|------|
| **商品信息管理** | 商品基础信息 | 维护商品通用信息：商品名称、SKU、条码、分类、主图、详情图、价格、单位、描述 |
| | 批量导入/导出 | 支持通过 Excel/CSV 批量导入商品信息，导出当前商品列表 |
| **平台信息同步** | 推送到平台 | 选择商品和平台，将商品信息（含图片）推送到对应平台，支持增量更新 |
| | 平台信息拉取 | 可从平台拉取现有商品信息到本地，用于初期迁移 |
| **同步状态与反馈** | 同步任务列表 | 每次同步生成任务记录，显示：发起时间、平台、商品数量、状态（进行中/成功/部分失败/失败） |
| | 详细失败信息 | 对于失败或部分成功，**逐条列出**失败商品、平台返回的错误码和错误描述（如“图片链接失效”、“SKU重复”） |
| | 重试机制 | 失败的同步任务可一键重试全部失败项，或单条重试，并支持设置最大重试次数 |

### 2.3 平台对接管理
| 功能模块 | 功能点 | 描述 |
|---------|--------|------|
| **平台授权** | 添加平台店铺 | 支持通过 OAuth 或 API Key 授权绑定平台店铺，目前支持：美团闪购、饿了么零售、京东到家、抖音小时达、微店（示例） |
| | 解绑/重新授权 | 可解绑，并显示 token 过期状态，提醒重新授权 |
| **平台配置** | 商品映射 | 将本地商品与平台商品建立映射关系（自动根据SKU匹配，也可手动修正） |
| | 同步策略 | 可针对不同平台设置：价格同步策略（按比例浮动或固定）、库存同步比例（如外卖平台留10%安全库存） |

### 2.4 系统管理与基础功能
| 功能模块 | 功能点 | 描述 |
|---------|--------|------|
| **用户与权限** | 多用户登录 | 支持店长、店员两种角色；店长可管理所有，店员仅限库存查看和手动调库（需审批） |
| | 操作日志 | 所有库存变动、商品信息更改、同步操作均有不可删除的日志 |
| **通知** | 失败通知 | 同步失败时，可配置通过站内信、邮件或企业微信/钉钉机器人通知 |
| | 低库存预警 | 安全库存线触发时标记提醒 |

## 3. 功能优先级 (MVP 及后续版本)

### 3.1 MVP（第一版，1-2个月）
**目标**：跑通核心链路，让一个便利店能用起来。

- 手动增/减库存（单商品、批量）
- 手动创建商品信息（名称、SKU、价格、图片上传）
- 实时库存看板（表格）及搜索
- 接入 **1个** 即时零售平台（如美团闪购）的 API：
    - OAuth授权绑定
    - 接收该平台订单自动减库存（通过轮询或平台推送）
    - 库存变更后自动同步库存量到该平台
- 同步失败详细反馈（显示失败原因）
- 基础操作日志

### 3.2 第二版（2-3个月后）
- 接入 2~3 个更多平台（饿了么、京东到家）
- 商品信息（主图、详情图）一键推送到平台
- 安全库存警戒提醒
- Excel 批量导入商品
- 简单的权限控制（店长/店员）

### 3.3 第三版（长期迭代）
- 多门店管理
- 盘点功能
- 共享库存/独占库存策略
- 通过扫码枪入库
- 更丰富的通知渠道
- 数据看板与销售分析（基于同步的订单数据）

## 4. 界面设计要求

### 4.1 设计原则
- **极简**：核心操作不超过3步，大量留白，避免密集数据压迫感，适合触屏平板操作（店主常用设备）。
- **状态强可视**：通过颜色（绿=正常，橙=同步中，红=失败/低库存）和图标让状态一目了然。
- **移动端适配**：优先设计桌面端（1024px+）和 iPad（768px）布局，手机端作为次要适配。

### 4.2 核心页面布局建议
1.  **左侧导航**（固定，宽200px）
    - 库存看板
    - 商品管理
    - 同步任务（带失败数量角标）
    - 平台授权
    - 系统设置

2.  **库存看板页**
    - 顶部：统计卡片（总商品数、低库存商品数、今日同步失败数）
    - 主体：可排序表格，列：商品名、SKU、当前库存、美团库存状态、饿了么库存状态… 每个商品行末有“加库存”“减库存”按钮。

3.  **商品管理页**
    - 左侧分类树，右侧商品表格，支持批量选择。
    - “推送到平台”按钮在下拉中列出已绑定平台，点击后弹窗确认。

4.  **同步任务页**
    - 任务列表，展开后可看到**每条失败详情**，并提供“重试”按钮。

### 4.3 色彩与组件
- 主色调：蓝绿色系（#0D9488），表现稳定、信任感。
- 状态色：成功#16A34A, 警告#EA580C, 错误#DC2626, 中立#3B82F6。
- 大量使用卡片、徽章(Badge)和标签(Tag)来展示状态。

## 5. 技术栈建议

基于“vibe coding”新手且使用 Claude Code，建议采用**前后端分离但整体尽可能用 TypeScript 全家桶**，上手快、生态好。

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **前端** | React + Next.js (App Router) + Tailwind CSS + shadcn/ui | 生态最丰富，shadcn/ui 提供大量可直接复制使用的组件，Claude Code 生成质量高；Next.js 可以做全栈，初期可不单独写后端 |
| **后端（API）** | Next.js API Routes 或独立 Node.js 服务器 (Hono/Express) | 如果业务逻辑不复杂，直接用 Next.js API 路由；后续若需独立扩展，可迁移至 Hono（超快，兼容 Web 标准） |
| **数据库** | PostgreSQL (可用 Supabase 托管) + Prisma ORM | PostgreSQL 支持事务、JSON，适合库存这种需要一致性的场景；Prisma 的 schema 声明式，Claude Code 极易理解 |
| **缓存** | Redis (Upstash) | 用于库存锁、平台 Token 缓存、实时库存数快速读取 |
| **消息队列** | 初期使用 BullMQ + Redis，或直接使用 Next.js 异步任务 | 订单到来时异步处理同步，避免阻塞；后续可换用 RabbitMQ |
| **文件存储** | AWS S3 / Cloudflare R2 | 存储商品图片，通过 CDN 加速 |
| **部署** | Vercel (前端+API) + Railway/Supabase (数据库) | 推送 Git 即自动部署，适合个人开发 |
| **监控** | Sentry + 自建日志 | 捕获同步失败和错误 |

**给 Claude Code 的提示**：逐步生成代码，可先建 Next.js 项目，用 shadcn/ui 搭布局，再写数据库模型，最后实现 API 和同步逻辑。

## 6. 非功能性需求

### 6.1 性能
- 库存看板首页（200个商品）加载时间 < 1.5秒。
- 订单自动减库存平均处理时间 < 500ms，且能正确处理并发订单（防超卖锁机制）。
- 商品图片上传需支持压缩和 CDN 加速，图片详情列表不卡顿。

### 6.2 可用性
- 系统正常运行时间 > 99.5%（除公告维护外）。
- 当某一平台 API 不可用时，不影响其他平台同步，并在界面上清晰标注该平台状态。

### 6.3 安全性
- 所有 API 请求需鉴权（JWT）。
- 平台 API Token / Secret 加密存储于数据库或环境变量，绝不明文落日志。
- 操作日志不可被删除或修改（仅追加）。
- 防止 CSRF/XSS，使用 Next.js 自带安全头和输入验证（zod）。

### 6.4 可扩展性
- 平台对接采用**插件化设计**，每个平台抽象出统一的 `syncInventory`, `syncProduct` 接口，新增平台只需实现接口。
- 数据库设计预留多门店字段（`store_id`），初期可为单门店模式。

### 6.5 数据一致性
- 扣减库存时使用数据库行锁或乐观锁，确保不发生超卖。
- 同步失败时，保留原始平台库存记录，待修复后可补偿同步。
